# 两阶段出题架构评估报告

## 用户问题澄清

**用户理解正确！** 现有逻辑确实存在这个问题：

### 场景举例

假设：

* 学员 Elo 分数：2501（对应 L4）

* 上传错题难度：L2

**现有逻辑**：

```
错题处理 → getTargetLevel(elo_score=2501) → 返回 'L4'
→ 生成 L4 难度题目（而非 L2）
```

**建议方案**：

```
错题处理 → 锁定错题本身的 level=L2
→ 生成 3 道 L2 难度题目（同难度三维度）
```

### 现有代码分析

```javascript
// 第 2281 行
const targetLevel = getTargetLevel(encounter.elo_score || 500)
// 这里使用的是 encounter.elo_score，而非错题本身的难度！
```

**问题根源**：错题的难度等级被 Elo 分数覆盖了。

***

## 建议方案的核心价值

### ✅ 架构优点

1. **职责分离清晰**

   * 阶段一：处理来源差异，统一上下文

   * 阶段二：纯粹的题目生成逻辑

   * 符合单一职责原则

2. **复用性强**

   * 阶段二完全复用，不关心来源

   * 新增来源类型只需修改阶段一

3. **错题场景特殊处理**

   * 通过 `errorPoint` 注入错题信息

   * 实现"隐性修复目标"

### ⚠️ 与现有系统的冲突

#### 现有系统架构

```
来源选择 → Elo分层 → 难度分配 → 生成题目
   ↓           ↓           ↓
active    Elo<1800    3道L2
error     1800-2500   1道L2 + 2道L3
bottom    Elo>=2501   1道L2/L3 + 1道L4 + 1道跨专项
```

#### 建议方案架构

```
来源解析 → 统一上下文 → 同难度三维度生成
   ↓           ↓            ↓
ERROR     variantId    Q1(计算) + Q2(逻辑) + Q3(结构)
MANUAL    + level
ELO_LOW   + factors
```

#### 核心冲突

| 对比项  | 现有系统                 | 建议方案           |
| ---- | -------------------- | -------------- |
| 难度分配 | Elo 动态分层（L2/L3/L4混合） | 同难度三维度         |
| 题目数量 | 3道（可能跨难度）            | 3道（同难度）        |
| 变例选择 | 自动匹配或用户指定            | 必须锁定 variantId |
| 高分学员 | L4挑战 + 跨专项综合         | 无跨专项逻辑         |

### 🔧 需要解决的问题

1. **Elo 分层逻辑如何保留？**

   * 建议方案的"同难度三维度"会丢失 Elo 分层的价值

   * 高分学员（Elo>=2501）需要 L4 挑战题

2. **因子复杂度分析如何实现？**

   * `analyzeFactorComplexity(factors)` 函数缺少具体实现

   * 需要定义"数值复杂度"的评分标准

3. **三种策略的定义过于抽象**

   * `CALC_HEAVY`：高复杂度数值，逻辑直球

   * `LOGIC_REVERSE`：低复杂度数值，逆向提问

   * `STRUCT_HIDDEN`：中等数值，隐含条件

   * AI 可能难以准确理解这些抽象概念

## 融合方案建议

### 方案：保留 Elo 分层 + 增强变式维度

```javascript
const generateUniversalTriad = async (context) => {
  const { level, factors, template, errorPoint, sourceType, elo } = context;

  // 1. 根据 Elo 决定难度分布（保留现有逻辑）
  let levelDistribution;
  if (elo < 1800) {
    levelDistribution = ['L2', 'L2', 'L2'];  // 全部 L2
  } else if (elo < 2500) {
    levelDistribution = ['L2', 'L3', 'L3'];  // 1道L2 + 2道L3
  } else {
    levelDistribution = ['L3', 'L4', 'L4'];  // 1道L3 + 2道L4
  }

  // 2. 为每个难度定义变式维度（新增逻辑）
  const strategies = [
    { type: 'CALC_HEAVY', twist: 'DIRECT', guide: '侧重运算稳健性' },
    { type: 'LOGIC_REVERSE', twist: 'REVERSE', guide: '侧重思维灵活性' },
    { type: 'STRUCT_HIDDEN', twist: 'HIDDEN_COND', guide: '侧重结构严谨性' }
  ];

  // 3. 并行生成
  const tasks = levelDistribution.map(async (targetLevel, i) => {
    const strat = strategies[i];
    // ... 生成逻辑
  });

  return await Promise.all(tasks);
};
```

### 关键改进点

1. **保留 Elo 分层**：难度分配仍由 Elo 决定
2. **增强变式维度**：同一难度内应用三种策略
3. **错题特殊处理**：注入 `errorPoint` 信息

## 结论

**建议部分合理，但需要融合现有 Elo 分层逻辑：**

| 保留           | 改进      |
| ------------ | ------- |
| Elo 分层规则     | 统一上下文解析 |
| 难度分配逻辑       | 变式维度策略  |
| 错题/手动/ELO 来源 | 错题点注入   |

**实施建议**：

1. 采用"阶段一：上下文解析器"的设计
2. 修改"阶段二"为"保留 Elo 分层 + 增强变式维度"
3. 先实现 Prompt 层面的变式约束，再考虑架构重构
