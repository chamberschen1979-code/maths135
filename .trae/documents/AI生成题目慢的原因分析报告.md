# AI 生成题目慢的原因分析报告

## 已优化项

| 优化项 | 状态 | 说明 |
|--------|------|------|
| 流式传输 | ✅ 已启用 | `stream: true` |
| Token 数量 | ✅ 已减少 | 从 8000 减少到 4000 |
| 并行生成 | ✅ 已使用 | `Promise.all` 并行生成 3 道题 |

---

## 可能导致慢的原因分析

### 原因 1：Prompt 过长（最可能）

**问题**：System Prompt 和 User Prompt 非常长。

**分析**：
- System Prompt 包含：
  - 核心使命
  - 难度分级宪法
  - 参考标杆（JSON 格式的样题）
  - 命题流程（5 个步骤）
  - 强制学术约束
  - 输出协议（XML 格式示例）
  - 错误示范和正确示范

- User Prompt 包含：
  - 母题信息
  - 参数池
  - 双难度分层策略
  - 因子锁定与回显规则
  - 解析绑定铁律
  - 严格要求
  - 双问制强制规则
  - 防污染指令
  - 绝对禁令

**估计 Prompt 长度**：
- System Prompt: ~2000 Token
- User Prompt: ~1500 Token
- 总计: ~3500 Token

**影响**：
- AI 需要更多时间解析和理解 Prompt
- 生成响应时需要遵循大量规则
- 每次生成都需要重新处理这些 Prompt

**优化方案**：
- 精简 Prompt，移除冗余内容
- 将示例移到外部文件
- 使用更简洁的规则描述

---

### 原因 2：API 服务器响应慢

**问题**：阿里云千问 API 服务器可能存在延迟。

**分析**：
- `qwen-plus` 模型推理时间较长
- 网络延迟
- 服务器排队

**影响**：
- 单次请求可能需要 10-30 秒
- 无法通过代码优化解决

**优化方案**：
- 使用更快的模型（如 `qwen-turbo`）
- 考虑使用其他 API 服务

---

### 原因 3：流式传输未实时更新 UI

**问题**：虽然启用了 `stream: true`，但代码仍然等待所有数据接收完毕后才处理。

**分析**：
```javascript
while (true) {
  const { done, value } = await reader.read()
  if (done) break
  // ... 累积所有数据
}
// 所有数据接收完毕后才处理
```

**影响**：
- 用户仍然需要等待所有数据接收完毕
- 流式传输的优势没有体现在 UI 上

**优化方案**：
- 实时更新 UI，显示正在生成的内容
- 使用状态管理逐步显示生成进度

---

### 原因 4：生成 3 道题的并行请求

**问题**：虽然使用了 `Promise.all`，但 3 个并行请求会同时消耗 API 资源。

**分析**：
```javascript
for (let i = 0; i < 3; i++) {
  problemPromises.push(generateAIProblem(...))
}
const results = await Promise.all(problemPromises)
```

**影响**：
- 3 个并行请求可能触发 API 限流
- 服务器可能降低响应优先级

**优化方案**：
- 考虑串行生成，但在 UI 上显示进度
- 或者减少生成题目数量（从 3 道减少到 2 道）

---

## 优化建议优先级

| 优先级 | 优化项 | 预期效果 | 实施难度 |
|--------|--------|----------|----------|
| 1 | 精简 Prompt | 时间减少 20-30% | 中 |
| 2 | 使用更快模型 | 时间减少 30-50% | 低 |
| 3 | 实时 UI 更新 | 感知速度提升 | 中 |
| 4 | 减少生成数量 | 时间减少 33% | 低 |

---

## 结论

**主要原因**：
1. **Prompt 过长**（~3500 Token）
2. **API 服务器响应慢**（`qwen-plus` 模型）
3. **流式传输未实时更新 UI**

**建议优先实施**：
1. 精简 Prompt 内容
2. 尝试使用 `qwen-turbo` 模型
3. 在 UI 上显示生成进度
